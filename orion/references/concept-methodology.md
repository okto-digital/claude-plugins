# Concept Methodology

How to build tiered website concepts from research and gap analysis outputs. Each concept describes what the client gets — not how it's implemented or what it costs. Implementation and pricing belong in the Proposal (Phase 6).

**Governing framework:** All concept decisions follow the Solution Framework — ICIP sequence (Interpret → Challenge → Invert → Propose), evidence-traced recommendations, simplicity preference, and honest labelling of CONFIRMED vs INFERRED conclusions.

---

## Concept vs Proposal boundary

| Concept answers | Proposal answers |
|---|---|
| What pages exist and what's on them | How long it takes to build |
| What the site looks and feels like | What tech stack delivers it |
| What the site says and how | What it costs per tier |
| What interactive features exist | Implementation options for the chosen concept |
| What tasks the project requires | Who does each task and at what rate |
| What the client can do independently | How ongoing support is structured |

**The concept is platform-agnostic.** It never says "WordPress" or "static HTML." It says "client can independently update all content pages" or "content updates require developer involvement." The proposal translates these capability statements into tech stack decisions with pricing.

**Why this separation matters:** The same concept can be delivered multiple ways at different price points. A 17-page site with 8 templates and a self-managing CMS could be WordPress custom, WordPress with premium theme, or Webflow. The concept is identical — the delivery approach and price differ. Mixing tech stack into the concept forces the client to evaluate platform choices alongside strategic choices, which muddies both decisions.

**The concept does state capability requirements** that constrain the proposal's tech choices:

| Concept states | Proposal interprets as |
|---|---|
| "Client independently manages all content" | CMS required (WordPress, Webflow, etc.) |
| "Content updates via agency on request" | Static build viable, retainer model |
| "Blog with regular publishing" | CMS with editorial workflow required |
| "Interactive portfolio filtering" | JavaScript framework or CMS with dynamic capabilities |
| "Booking system integrated" | Third-party integration or custom development |
| "Multi-language content with independent updates per language" | CMS with multilingual plugin/architecture |

## The tier model

Three tiers. Each tier is a **genuinely different business proposition** — not the same site with features added or removed. They differ in strategic ambition, and every tier must deliver a measurably better result than the client's current website.

**Tier 1 — Efficient**

A focused, strategic intervention. Fixes what's broken, captures the most critical opportunities, delivers a professional result. May preserve and optimise the existing structure rather than rebuilding. The client invests less and gets a site that works — it stops losing business but doesn't aggressively compete for new market share.

Business fit: budget-constrained, fast timeline, pragmatic, "fix and move on."

**Tier 2 — Competitive**

The recommended approach. Restructures the site based on research — keyword opportunities, competitor gaps, persona journeys. Every page exists for a research-backed reason. The client invests in a site specifically designed to outperform identified competitors on identified opportunities.

Business fit: growth-oriented, willing to invest in competitive advantage, wants measurable results.

**Tier 3 — Dominant**

The full vision. Captures every opportunity the research identified, including whitespace and emerging trends. Positions the client ahead of competitors, not just at parity. Requires ongoing content commitment and higher investment but builds a long-term market position.

Business fit: market leader ambition, has content/marketing resources, sees the website as the primary business development tool.

**The floor rule:** Even Tier 1 must produce a site that is demonstrably better than the client's current website on every dimension the research identified as problematic. If the current site has 61/100 accessibility, Tier 1 must improve it. If the current site has zero trust signals, Tier 1 must add them. The tier choice affects *how much better* — never *whether better*.

## Modifier dimensions

The tiers don't just differ by "more or fewer pages." They differ across 8 specific dimensions, each independently adjustable.

### 1. Structure (pages and hierarchy)

What pages exist, how they're organised, and what purpose each serves.

| Tier 1 | Tier 2 | Tier 3 |
|---|---|---|
| Pages addressing CRITICAL gaps only. Structural pages (home, contact, legal) + core service pages. If redesigning, assess whether existing structure can be preserved and optimised rather than rebuilt. | Pages addressing CRITICAL + COMPETITIVE gaps. New pages where research shows opportunity. Full restructure based on keyword clusters and persona journeys. | All of Tier 2 + WHITESPACE opportunities. Aspirational keyword targets. Full blog/editorial section. Advanced page types (tools, calculators, portals) if market supports them. |
| Fewer pages, flatter hierarchy | Research-driven page count, logical hierarchy | Full page set, deep hierarchy with content clusters |

### 2. Templates (layout complexity)

How many distinct page layouts are designed and built, and how sections distribute across them.

| Tier 1 | Tier 2 | Tier 3 |
|---|---|---|
| Minimise template count. Merge where structurally possible. If pages can share a layout without compromising purpose, they should. Reuse existing templates if redesigning and they're structurally sound. | One template per genuinely distinct page purpose. The wireframe test: if the skeletons differ, the templates differ. Shared templates only where pages genuinely share structure. | Unique layouts where differentiation justifies it. Template variations for special content (case study detail, interactive comparison, rich media pages). |
| Fewer templates, more template reuse | Honest template count based on structural reality | Extended template set with specialised variations |

### 3. Depth (sections per template)

How many content sections each template contains and how substantial each section is.

| Tier 1 | Tier 2 | Tier 3 |
|---|---|---|
| Essential sections only per template. Hero + core content + CTA. Strip supporting sections (FAQ, testimonials, secondary blocks) unless they directly address a CRITICAL gap. | Full section set based on competitor analysis and persona needs. Every section earns its place by serving a research-identified purpose. No section exists without justification. | Tier 2 sections + differentiating sections. Interactive elements, multimedia blocks, advanced trust signals, content blocks competitors don't have. Richer page experiences. |
| 3-4 sections per template | 5-7 sections per template | 6-9 sections per template |

### 4. Content quality

How deep, polished, and strategically crafted the content is — across all tiers, the voice direction stays the same. Voice quality never degrades.

| Tier 1 | Tier 2 | Tier 3 |
|---|---|---|
| Functional. Serves the keyword cluster and conversion flow. Answers the persona's primary question. Minimum viable content per page — but still professionally written with the correct voice. | Competitive. Matches or exceeds what top competitors provide. Serves the persona at the relevant journey stage. Incorporates messaging intelligence from research — real customer language. | Leading. Sets the standard in the market. Original content formats. Deeper coverage than any competitor. Editorial quality with data, examples, and original insight. |

### 5. Visual ambition

How far the design pushes from industry conventions. Even Tier 1 must look professional and current.

| Tier 1 | Tier 2 | Tier 3 |
|---|---|---|
| Industry-aligned. Follows conventions identified in research. Clean, modern, professional. The goal is "competent and current" — no longer dated, not trying to stand out. If redesigning, preserve brand elements that work while modernising what's dated. | Differentiated on 1-2 dimensions identified in gap analysis. A distinctive colour, imagery style, or typography choice that's grounded in research (persona needs, market gap, competitor weakness). Recognisable within the market but distinct. | Market-leading. Draws from global references outside the local competitive field. Bolder creative direction. May incorporate emerging visual trends that local competitors haven't adopted. Higher design investment justified by research. |

### 6. Functionality

Interactive features that create user-site engagement beyond default browsing. Strictly scoped — performance, accessibility, SEO, and analytics are NOT functionality.

| Tier 1 | Tier 2 | Tier 3 |
|---|---|---|
| Must-have interactive features only. Contact form + basic gallery (if relevant). Whatever the minimum is for this site type to function. | Must-have + competitive features. Enhanced form (service routing, conditional fields). Gallery with lightbox and navigation. Blog with filtering. Features direct-threat competitors have that the client doesn't. | Tier 2 + differentiating features. Booking systems, portals, interactive tools, advanced search/filtering, client dashboards — features from emerging trends or whitespace analysis. |

**Classification test:** Does this involve the visitor doing something and the site responding?

| IS functionality | Is NOT functionality |
|---|---|
| Contact form (visitor submits, site validates and sends) | Cookie consent (compliance task) |
| Portfolio gallery with lightbox (visitor clicks, site responds) | WCAG compliance (standard/quality requirement) |
| Blog category filtering (visitor selects, site filters) | GA4 setup (analytics task) |
| Search (visitor types, site returns results) | Image optimisation (performance task) |
| Booking calendar (visitor selects date, site checks availability) | Schema markup (SEO task) |
| Quote calculator (visitor inputs, site calculates) | 301 redirects (migration task) |

### 7. Things to do (project scope)

The concrete deliverables — tasks, migrations, setups, compliance items. Split into included (inseparable from the build) and optional (separately priced or done by client).

| Tier 1 | Tier 2 | Tier 3 |
|---|---|---|
| Core build + compliance + migration. Minimal included set. More items pushed to optional. Content services almost always optional. Marketing services always optional. | Full professional deliverable set. Balanced included/optional split. Core build + compliance + migration + performance + SEO foundation included. Content and marketing services optional. | Extended included set. Content production, editorial setup, advanced integrations, and ongoing optimisation pulled into included. Fewer optional items — more is part of the package. |

**Included** = inseparable from delivering the website. Test: would the site be broken, non-compliant, or non-functional without this?
**Optional** = adds value but the website works without it. Test: can we launch without this?
**Client responsibilities** = items only the client can provide. Not priced.

### 8. Client independence

How much the client can do without agency involvement after launch. This is the capability dimension that constrains the proposal's tech stack choice.

| Tier 1 | Tier 2 | Tier 3 |
|---|---|---|
| May range from "client manages all content" to "agency manages updates on request" depending on client capability and budget. The concept states the capability; the proposal determines the implementation. | Client independently manages content, adds blog posts, updates service pages, manages gallery content. Agency involvement only for structural changes or new features. | Full editorial independence. Client manages content, publishes on schedule, manages multilingual variants if applicable. Self-service analytics dashboards. Agency involvement only for strategy reviews and technical upgrades. |

## The structure decision process

The sitemap is the most critical output. Every other dimension depends on it. The process must be rigorous because wrong structure = wrong everything downstream.

**Apply ICIP before building the sitemap:**

**INTERPRET:** What does the combined research show about what pages this site needs? Look at keyword clusters (what people search for), persona journeys (what path they follow), content landscape (what competitors have for each cluster), client inventory (what exists today), and confirmed gap analysis decisions. State the dominant pattern in one sentence before proposing any pages.

**CHALLENGE:** What complicates this? Are there keyword clusters that suggest pages the client shouldn't have? Are there persona journey stages with no keyword evidence? Does the client inventory show pages that rank well but don't fit the new structure?

**INVERT:** What would make this site structure fail? A structure that doesn't match how the target persona navigates. Pages that target keywords the client can't realistically rank for. Missing a page that every competitor has (table-stakes). Creating pages with no keyword target and no structural purpose. Cannibalisation — two pages competing for the same cluster.

**PROPOSE:** The simplest structure that addresses the confirmed findings, avoids the failure conditions, and is honest about what's inferred.

### Building the sitemap

**Step 1 — Start from keyword clusters.** Every page that targets organic traffic must map to a keyword cluster. Take the clusters and their page type recommendations: core_service, supporting_content, local_variation, comparison, aspirational.

**Step 2 — Add structural pages.** Pages that serve the user journey or business requirements but don't target a specific keyword cluster: Homepage, About, Contact, Process, Pricing, Blog index, Legal/utility. For each structural page, the justification must be explicit — "every website needs X" is not a justification.

**Step 3 — Cross-reference against client inventory.** For redesigns, map every proposed page against the existing inventory. Proposed page has existing equivalent → assess: rewrite or optimise? No existing equivalent → create new. Existing page has no proposed equivalent → redirect or drop. Check keyword baselines: does the existing page rank for anything?

**Step 4 — Validate against personas.** Walk each persona's journey through the proposed sitemap. Awareness → Consideration → Decision. Does the primary persona reach their goal in 3 clicks or fewer from homepage?

**Step 5 — Check for cannibalisation.** Does each keyword cluster map to exactly one page? If two pages target the same cluster, one must change.

**Step 6 — Apply the tier modifier.** Tier 1: CRITICAL-classified clusters + structural necessities + legal. Tier 2: CRITICAL + COMPETITIVE clusters + full structural set. Tier 3: All of Tier 2 + WHITESPACE + aspirational + editorial.

**Step 7 — Produce the sitemap.** Tree showing hierarchy, URLs, keyword targets, and search volume per page. Plus template mapping and complexity summary.

## Template identification

**The wireframe test:** Remove all text and images from two pages. Are the remaining skeletons identical? Same number of sections, same section types, same layout pattern? If yes → same template. If no → different templates.

**Template reuse is valid when:**
- Pages genuinely share the same section structure
- Content varies but layout doesn't
- The template supports content variants through conditional sections

**Template reuse is invalid when:**
- Pages have different section counts
- Pages have different section types
- Pages serve fundamentally different user tasks
- Forcing reuse would require awkward compromises

**Tier modifier on templates:**
- Tier 1: Actively seek valid reuse. Can pages share a layout if stripped to essentials? For redesigns: can existing templates be preserved?
- Tier 2: Honest count. Don't force reuse where it doesn't work, don't create unique templates where reuse is valid.
- Tier 3: Allow specialised variations where the content structure genuinely differs. Each variation must be justified.

## The concept-proposal handoff

The concept produces 3 concept documents + a comparison summary.

**What the concept hands off to the proposal per dimension:**

| Concept dimension | Proposal translates into |
|---|---|
| Structure (sitemap) | Page count → effort estimation per template |
| Templates (with sections) | Template count x section count → design + development hours |
| Visual direction | Design phase scope → design hours, revision rounds |
| Content strategy | Content production scope → copywriting hours per page |
| Functionality | Feature list → development hours per feature |
| Things to do (included) | Line items → task hours, third-party costs |
| Things to do (optional) | Add-on pricing menu |
| Client independence requirements | Tech stack selection → platform licensing, hosting costs |
